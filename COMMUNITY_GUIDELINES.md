# Community Guidelines

## Overview

Welcome to the Haspen UI Design System community! These guidelines help ensure our community remains
welcoming, productive, and focused on building high-quality UI components for Danish web
applications.

## Code of Conduct

### Our Standards

We are committed to providing a friendly, safe, and welcoming environment for all, regardless of:

- Experience level
- Gender identity and expression
- Sexual orientation
- Disability
- Personal appearance
- Body size
- Race
- Ethnicity
- Age
- Religion
- Nationality

### Expected Behavior

- **Be respectful**: Treat everyone with respect and kindness
- **Be collaborative**: Help others learn and grow
- **Be constructive**: Provide helpful feedback and suggestions
- **Be patient**: Remember that everyone has different experience levels
- **Be inclusive**: Use welcoming and inclusive language

### Unacceptable Behavior

- Harassment, discrimination, or inappropriate conduct
- Trolling, insulting comments, or personal attacks
- Publishing private information without consent
- Any conduct that would be inappropriate in a professional setting

## Development Guidelines

### Quality Standards

- Follow the atomic design principles outlined in `CLAUDE.md`
- Write comprehensive tests for all components (minimum 90% coverage)
- Include proper TypeScript interfaces for all props and events
- Document components with Storybook stories and README files
- Follow the established code style and linting rules

### Danish Localization

- Prioritize Danish language support in components
- Include proper Danish formatting for dates, currency, and phone numbers
- Ensure CPR number validation follows Danish standards
- Consider Danish accessibility requirements (WCAG compliance)

### Contribution Process

1. **Discussion First**: For significant changes, open an issue to discuss your approach
2. **Branch Naming**: Use descriptive branch names (e.g., `feat/button-variant`,
   `fix/form-validation`)
3. **Commit Messages**: Follow conventional commits format (`type(scope): description`)
4. **Pull Requests**: Include clear description, testing instructions, and screenshots if applicable
5. **Code Review**: Be open to feedback and iterate based on suggestions

## Communication

### Issue Reporting

- Use the provided issue templates for bug reports and feature requests
- Include reproduction steps, expected behavior, and actual behavior
- Add relevant labels and assign to the appropriate milestone
- Search existing issues before creating new ones

### Discussions

- Use GitHub Discussions for questions, ideas, and general community interaction
- Tag posts appropriately (`question`, `idea`, `showcase`, etc.)
- Be specific in your titles and provide context in your posts

### Getting Help

- Check the documentation in `CLAUDE.md` and component README files first
- Search existing issues and discussions
- Ask specific questions with code examples when possible
- Be patient waiting for responses - maintainers contribute in their free time

## Enforcement

Community guidelines violations may result in:

1. **Warning**: First offense or minor violations
2. **Temporary Ban**: Repeated violations or moderate offenses
3. **Permanent Ban**: Severe violations or continued problematic behavior

Reports can be made by contacting the maintainers directly or opening a confidential issue.

## Recognition

We value all contributions to the Haspen UI community:

- Code contributions (components, fixes, improvements)
- Documentation improvements
- Bug reports and testing
- Community support and mentoring
- Design feedback and suggestions

Contributors will be recognized in our changelog and contributor list.
